# Chunk 6 — Decision bloodline + checkpoints

> Brief reworked after Chunk 1's completion (2026-07-04); supersedes the
> original scaffold brief.

You are implementing Chunk 6, the final chunk of Galapagos. Read
`docs/vision.md` and `docs/architecture.md` first — they are the binding
contract and override this brief on any conflict.

## State when you start

Chunks 1–5 done: Darwin chat, per-project records (decision validation
already enforced in `core/records/schema.ts`, no tagging yet), workers/lanes
with worktrees under `<GALAPAGOS_STATE_DIR>/worktrees/`,
monitoring/evidence/confidence, clarity + digests. Decision records — like
all records — live in each target repo's `docs/galapagos/`, and checkpoint
tags belong to the **target repo's** git history.

## Build

- Checkpoint mechanism (architecture §8) triggered by
  `write_record(type=decision)`:
  1. target ref = relevant worker worktree HEAD or the project's main HEAD;
  2. dirty worktree → WIP commit `galapagos: pre-decision checkpoint <id>`;
     on failure `git_checkpoint_status: blocked_dirty` + attention item —
     never silently skip;
  3. `git tag galapagos/decision/<shortid> <sha>` in the target repo (extend
     `adapters/git/` mutating surface by exactly commit/tag here — keep the
     same narrow-allowlist style as init and worktree operations);
  4. record written with `git_checkpoint_ref` + `parent_decision_ref` =
     current tip decision of that line; record file committed under the
     Chunk 2 narrow-staging rules.
- `core/decisions/tree.ts`: pure graph builder — decision records →
  nodes/edges via `parent_decision_ref`, forks visible.
- `/decisions` page — **the bloodline**, its own page: the active project's
  decision lineage as a graph; each node shows title, chosen_path, date,
  checkpoint status; calm presentation, the project's history readable at a
  glance.
- Resume-from-node: node action → `git worktree add
  <GALAPAGOS_STATE_DIR>/worktrees/<project-slug>/resume-<shortid> -b
  resume/<slug> galapagos/decision/<shortid>` + new lane (+ worker if
  requested) + child decision record pointing at the resumed node, so the
  bloodline forks visibly.

## Forbidden

No force-pushes, no tag deletion, no history rewriting, no decisions table in
SQLite (the tree derives from records only), no worktrees inside target
repos, no auto-decisions — every decision node originates from the user or an
explicitly user-approved manager proposal.

## Exit criterion

User makes a decision in chat → tag exists in the target repo, node appears
in the bloodline; user resumes from an earlier node → a new worktree/branch
materializes under the state dir, a worker starts there, and the bloodline
shows the fork.

## Verification

- `npm test` green: checkpoint integration tests on a fixture repo
  (dirty-tree, clean-tree, tag-collision), pure tree-builder tests.
- Manual full resume drill (user, own terminal) as in the exit criterion.
- Final overall drill (all chunks): state a goal, spawn two non-overlapping
  workers, let one go stale, answer a triaged question, make a decision,
  resume from an earlier decision node — every step observable in the UI with
  sourced data, and every target repo left clean apart from
  `docs/galapagos/` and its own checkpoint history.
