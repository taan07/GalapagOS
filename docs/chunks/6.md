# Chunk 6 — Decision bloodline + checkpoints

You are implementing Chunk 6, the final chunk of Galapagos. Read
`docs/vision.md` and `docs/architecture.md` first — they are the binding
contract and override this brief on any conflict.

## State when you start

Chunks 1–5 done: chat, records (decision validation already enforced, no
tagging yet), workers/lanes, monitoring/evidence/confidence, clarity + digests.

## Build

- Checkpoint mechanism (architecture §8) triggered by
  `write_record(type=decision)`:
  1. target ref = relevant worker worktree HEAD or project HEAD;
  2. dirty worktree → WIP commit `galapagos: pre-decision checkpoint <id>`;
     on failure `git_checkpoint_status: blocked_dirty` + attention item —
     never silently skip;
  3. `git tag galapagos/decision/<shortid> <sha>` (via
     `adapters/git/mutating-runner.ts` — commit, tag, worktree add only);
  4. record written with `git_checkpoint_ref` + `parent_decision_ref` = current
     tip decision of that line; record file committed.
- `core/decisions/tree.ts`: pure graph builder — decision records →
  nodes/edges via `parent_decision_ref`, forks visible.
- `/decisions` page — **the bloodline**, its own page: the lineage of every
  major decision as a graph; each node shows title, chosen_path, date,
  checkpoint status; calm presentation, the project's history readable at a
  glance.
- Resume-from-node: node action → `git worktree add
  .galapagos/worktrees/resume-<shortid> -b resume/<slug>
  galapagos/decision/<shortid>` + new lane (+ worker if requested) + child
  decision record pointing at the resumed node, so the bloodline forks
  visibly.

## Forbidden

No force-pushes, no tag deletion, no history rewriting, no decisions table in
SQLite (the tree derives from records only), no auto-decisions — every decision
node originates from the user or an explicitly user-approved manager proposal.

## Exit criterion

User makes a decision in chat → tag exists, node appears in the bloodline;
user resumes from an earlier node → a new worktree/branch materializes, a
worker starts there, and the bloodline shows the fork.

## Verification

- `npm test` green: checkpoint integration tests on a fixture repo (dirty-tree,
  clean-tree, tag-collision), pure tree-builder tests.
- Manual full resume drill as in the exit criterion.
- Final overall drill (all chunks): state a goal, spawn two non-overlapping
  workers, let one go stale, answer a triaged question, make a decision, resume
  from an earlier decision node — every step observable in the UI with sourced
  data.
