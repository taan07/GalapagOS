Operational facts (paid for in Chunk 1 — do not rediscover):

- **Auth is keychain-bound.** Every session spawn goes through the shared
  helper that sets `pathToClaudeCodeExecutable` to the user's installed
  binary (`config.claudeBinPath`); live sessions only authenticate when the
  daemon runs from the user's own terminal. Build/`npm test` anywhere; the
  user verifies live worker runs.
- **cwd is load-bearing**: manager sessions pin the project root; worker
  sessions pin their worktree. Never spawn without an explicit cwd.
- Tests: `tsconfig.node.json` → `dist-node`, `node --test`. Boundaries:
  `core/` pure, `adapters/` I/O, UI talks only to route handlers.

## Build

- Tables + repos: `lanes`, `workers`, `worker_events`, `completion_digests`,
  `attention_items` (architecture §3; all project_id-scoped like everything
  else in the central DB).
- `core/lanes/lane-check.ts`: pure — (changed file list × allowed/forbidden
  picomatch globs) → violations. Exhaustive glob-matrix unit tests.
- Worktrees: created under
  `<GALAPAGOS_STATE_DIR>/worktrees/<project-slug>/<lane-slug>/` (architecture
  §6 — NEVER inside the target repo). Extend the mutating-git surface only as
  far as `worktree add`/`worktree remove` for these paths.
- Manager tools: `spawn_worker(lane, brief, model?)` (creates lane row —
  rejecting allowed-glob overlap with any active lane in the same project —
  plus a `worker_brief` record in the target repo, the worktree, and the
  worker session via the daemon; `model` defaults to
  `GALAPAGOS_WORKER_MODEL`, default `claude-fable-5`),
  `steer_worker(id, message)`, `stop_worker(id)`, `list_workers`,
  `worker_status(id)`.
- Worker sessions (`src/adapters/agent/worker-session.ts`): implement behind
  the harness-agnostic interface from architecture §6 (Agent SDK is the only
  backend in this chunk): streaming-input `query()` with `cwd` = the
  worktree, spawned via the shared session helper; daemon persists every
  streamed message to `worker_events` and updates
  `last_message_at`/`last_summary`; `canUseTool` denies Edit/Write outside
  lane globs with an explanatory message (preventive layer only — Bash bypass
  is expected and handled by the detective audit; the full audit loop is
  Chunk 4, but run the lane-check once on worker stop in this chunk).
- Completion report contract (architecture §6): worker system prompt requires
  the fenced `galapagos-completion` JSON; daemon parses into
  `completion_digests`; missing/malformed → `unstructured_completion`
  attention item and the worker is not rendered done. (Attention queue UI is
  Chunk 4; the table and rows land now.)
- `/workers` page: worker list (lane, status, liveness) + drilldown with live
  event stream, matching the existing UI conventions (dark calm CSS variables
  in `globals.css`, chips for tool events, explicit error/empty states).
  Introduce the app's quiet top nav here if `/records` hasn't already forced
  it.

## Forbidden

No monitor interval loop, no confidence engine, no auto-triage, no evidence
runner, no editing of the karz98rk reference repo, no shared files between
lanes, no worktrees inside target repos, no worker without a lane contract.

## Exit criterion

Via chat in a real project, the user spawns a worker on a scoped task in its
own worktree, watches it stream in `/workers`, steers it mid-run; a deliberate
out-of-lane edit made via Bash is caught at worker stop and raises a
`lane_violation` attention row.
