## State when you start

Chunks 1–4 built: Darwin chat with per-project records + distillation; workers
in lanes with parsed completion digests; the monitor (30s, zero-LLM);
evidence runs keyed to workspace state; the PURE four-leg confidence engine
(facts, tripwires, watchdog, critic — every signal/cap names its leg; a
completion no independent leg has reviewed is never strong or auto-reviewed);
leg verdicts persisted as jobs rows that re-key on what they judged;
event-driven triage on a fresh seeded session; the attention queue
(integrity_alert included) with resolve/dismiss on `/`; gauges with
leg-grouped "why" drilldowns on `/` and `/workers`. UI style: dark calm
mission control (CSS variables in `globals.css`, expandable chips, explicit
empty/error states, UI never imports adapters).

## Build

- Clarity view-model (`core/clarity/view-model.ts`), written fresh and pure:
  assembles the clarity answers from records + live signals. Non-negotiable
  pattern: **every displayed field carries source attribution** (source,
  sourceLabel, sourceRecords) and missing data renders as explicitly missing
  with source "missing" — never fabricated, never defaulted. Unit tests must
  cover both the attribution of every field and the missing-data rendering.
- Per-worker clarity: **now** (last_summary + live `git diff --numstat` vs
  `base_sha` + latest evidence) vs **inception** (worker_brief record + lane
  contract).
- **Per-worker clarity panel beside the confidence gauge** (the coupling,
  side 1): brief coverage — which agreed specifics (user_answer records) the
  worker's brief actually cites, and which open_question records overlap its
  lane — rendered next to the gauge on /workers so a vibes-brief is visible
  as structural risk BEFORE any code lands.
- **Clarity as a worker-level confidence input** (facts leg): a brief that
  cites recorded specifics raises trust capacity; unresolved lane-relevant
  open questions lower it. This is a §9 semantics extension: it ships ONLY
  with new gold scenarios confirmed with the user in-session (the question
  channel works — use it), added to tests/confidence-engine.test.ts first.
- **Understanding-gap feedback** (the coupling, side 2): a fresh critic
  rejection or contradicted claim renders in the clarity view as an
  understanding-gap row with its evidence linked (the critic finding, the
  contradicting run). The durable record for a confirmed gap is an
  open_question — no new record types.
- Project clarity: latest synthesis + active goal (records) vs aggregate
  worker state and open attention.
- `/clarity` page rendering both, calm and quiet.
- Completion digest UI as the lead surface of a finished worker's drilldown,
  in the four vision layers, in order:
  1. tiny narrative (≤3 sentences),
  2. before/after in product terms,
  3. visual change map — touched areas sized by change magnitude (numstat),
     colored in-lane / tested / untested / out-of-lane,
  4. claim checklist with verified/unverified/contradicted evidence badges,
     proof one click away (evidence run, diff, file) — consume the evidence
     adapter's LinkedClaim output (served via /api/confidence claimLinks);
     do not invent a second claim-linking.
  Raw transcript and full diffs exist only behind explicit drilldowns.
- `core/digests/assemble.ts`: pure assembly of digest layers from
  completion_digests + numstat + lane + evidence rows. Unit-tested.

## Forbidden

No fabricated or default-filled fields; no raw agent prose rendered by default
anywhere; no new tables or record types; no confidence semantics changes
EXCEPT the worker-level clarity input above, and only via new §9 gold
scenarios confirmed with the user first.

## Exit criterion

Every displayed field on `/clarity` is traceable to a named record or live
signal; a finished worker's result is absorbable from the digest alone —
narrative, before/after, change map, badges — without reading transcript
text; and a worker briefed on vibes visibly differs (clarity panel, and gauge
once the §9 extension is confirmed) from one briefed against records, before
any code is written.

## Verification

- `npm test` green: view-model unit tests (source attribution, missing-data
  rendering), digest layer assembly tests, and — if the §9 extension ships —
  the new clarity gold scenarios.
- Manual: cross-check three displayed fields against their cited sources.
- Manual (user, own terminal): run a real worker to completion and absorb the
  result from the digest only; spawn one worker with a vibes-brief and one
  briefed against records, and see the difference on /workers before either
  finishes.
