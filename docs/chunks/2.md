Facts from the finished chunk that will save you hours:

- **Auth is keychain-bound.** Sessions spawn the user's installed binary
  (`config.claudeBinPath` → `pathToClaudeCodeExecutable`, already wired) and
  live turns only authenticate when the daemon runs from the user's own
  terminal. Build and `npm test` anywhere; the user verifies live turns.
- **Resume is cwd-keyed** — every `query()` passes the project root as `cwd`.
  Preserve this in anything you add (including distill forks).
- Tests compile via `tsconfig.node.json` → `dist-node`, run with `node --test`
  (`npm test` = typecheck + suites). Module boundaries are enforced by
  convention: `core/` pure, `adapters/` I/O, UI never imports adapters.
- Manager tools live in `src/adapters/agent/manager-tools.ts` and receive a
  `ManagerToolContext` (project root/slug, vault path, event emitter) — extend
  that context rather than importing config inside tools.

## Build

- `core/records/frontmatter.ts`: written fresh — strict, dependency-free YAML
  frontmatter parse/render for exactly the shapes records use (quoted strings
  with escaping, string arrays, nulls; reject unknown structure loudly).
  There's a working in-repo reference for the flavor:
  `src/adapters/vault/specifics.ts` already writes/parses this frontmatter
  style. Full round-trip tests.
- `core/records/schema.ts`: the 8 `glp_type` record types + validation rules
  (architecture §4). Decision records validate `decision_options`,
  `rollback_note`, `confidence_impact`; `chosen_path` required before closed
  status. (Checkpoint git-tagging itself is Chunk 6 — validation only here.)
- `adapters/records/store.ts`: written fresh to the architecture §4 contract —
  read/index all records; create with the wx flag (never overwrite), per-type
  subdirectories, `YYYY-MM-DD-<slug>-<shortid>.md` filenames; update limited
  to status transitions (open statuses on create; closed statuses —
  resolved/done/approved/superseded/archived — only via update) and appended
  notes. Full create/update/validation test coverage. **Records are
  per-project, in the TARGET repo** (`<project.root_path>/docs/galapagos/…`)
  — never in the galapagos repo itself (architecture §5).
- Wire real `read_records` / `write_record` / `update_record` manager tools
  (replace the stub; keep its honesty — an empty store reports "no records
  yet", never invented content).
- **`record_specific` becomes a write-through:** it now creates a
  `user_answer` record in the store AND keeps writing the vault mirror file,
  so the Obsidian view and the specifics panel keep working unchanged. One
  memory, two views — never two diverging memories.
- **Ingest existing vault specifics, idempotently.** On store initialization
  per project, import agreed-specific files (glp_type `agreed_specific`) into
  `user_answer` records, marking each vault file (e.g. `migrated_to:
  <record-id>` in frontmatter) so re-runs never duplicate. Darwin's memory
  must not reset or fork when the records store arrives.
- Distillation per architecture §5: post-turn `distill` job (jobs table), run
  on a **fork** of the manager session with `GALAPAGOS_DISTILL_MODEL`
  (default `claude-haiku-4-5`); mark covered turns `distilled_at`;
  auto-commit `docs/galapagos/` in the target repo with message prefix
  `galapagos(records):` under the narrow-staging safety rules
  (architecture §5 — only `docs/galapagos/` paths, skip + surface on failure).
- Compact-by-re-brief: when resume fails or context is bloated, open a fresh
  session seeded from latest synthesis + active goal + open questions +
  recent user answers; mark old session `compacted`. This replaces Chunk 1's
  bare "re-briefed" note with a real record-seeded recovery.
- `/records` page: plain browser of the active project's records (type,
  title, status, date, body), every field source-attributed. No editing UI.

## Forbidden

No git tags/checkpoints, no workers, no monitoring, no new SQLite tables, no
record types beyond the 8, no transcript dumps into records (doctrine only),
no distillation on Darwin's main session (forks only), no staging of any
target-repo path outside `docs/galapagos/`.

## Exit criterion

In the active project, the user states the project goal in chat → a committed
`docs/galapagos/goals/*.md` `active_goal` record exists in that project's
repo; pre-existing vault specifics appear as records without duplication on
restart; force a fresh session (delete the sdk session id) → Darwin recalls
the goal via record-seeded re-brief, visibly.
