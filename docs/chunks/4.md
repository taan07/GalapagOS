>    *Implication:* the checkout-ui worker says "done" → its gauge is

> 3. **Triage runs on a fresh seeded session (user-confirmed).** Seed =

>    widen its surface (the manager/distill niggle does NOT extend to

>    *Implication:* triage works even if you never chatted, and never
>    contends with Darwin's busy flag — but it won't know conversational
>    nuance that was never distilled into records (that is what records
>    are for).
> 4. **Main-checkout watch: fingerprint deltas per tick
>    (user-confirmed).** While workers are live, NEW dirty files in the
>    project's PRIMARY checkout that fall inside a live lane's globs
>    raise ONE deduplicated `lane_violation` (normal priority) worded
>    "possibly your own edit". Files no lane claims are ignored; the
>    baseline resets on daemon restart.
>    *Implication:* a payments worker running `echo hack >>
>    ~/Dev/web-shop/src/api/payments/stripe.ts` via Bash with an absolute
>    path is caught within 30s — while you editing your own
>    `docs/plan.md` mid-run pages nobody. You editing
>    `src/checkout/*.tsx` while the checkout worker is live DOES raise
>    one (worded) item — the price of closing the hole.
> 5. **Evidence keys.** `evidence_runs.head_sha` stores `<sha>` for a
>    clean tree, `<sha>+dirty.<fingerprint16>` for a dirty one; freshness
>    is string equality against the workspace's current key.
>    *Implication:* ANY commit or uncommitted edit after a run makes its
>    evidence stale — the gauge starts draining on the next tick, and
>    "tests passed (before I kept coding)" can never render as health.
> 6. **`worker_failed` attention kind added** (§3 comment updated in the
>    same commit, the chunk-3 `steer` precedent). Raised at the moment a
>    session dies MID-RUN; spawn failures stay off the queue because they
>    return synchronously to whoever asked.
>    *Implication:* a worker whose session hits an auth/model error at
>    2am is on your queue at 8am with the worktree path in the detail —
>    previously it only blocked the gauge and nobody was told why.
> 7. **Silence thresholds** (`GALAPAGOS_STALE_WORKER_SECONDS`, default
>    300): running/spawning silent past it → `stale_worker` (high), once
>    per silence episode; `awaiting_input` past it → `question_for_user`
>    (high) carrying the worker's last words. This reconciles the chunk 3
>    ruling (a fresh question is dialogue, not an exception) with this
>    chunk's exit criterion (a question must reach the user via triage):
>    only a question left hanging escalates.
>    *Implication:* the checkout worker asks "Apple Pay too?" — for five
>    minutes that is Darwin's to relay; if nobody does, it becomes an
>    attention item and triage carries it into your chat with a
>    recommendation.
> 8. **Zero-LLM auto-review.** The monitor marks a digest
>    `manager_reviewed` only when the worker's confidence report is
>    `strong` AND it has no open attention — which by construction means
>    fresh passing required checks, verified claims, clean lane audit.
>    Anything less stays `parsed`, raises items, and goes to LLM triage.
>    *Implication:* clean completions appear as done with zero tokens
>    spent; nothing is ever rubber-stamped on prose.
> 9. **Monitor dedup discipline.** Identical OPEN facts are never
>    re-raised (a changed fact — new violating file, new episode — is);
>    the stop audit dedupes against the monitor's identical open item;
>    `unsupported_claim` items auto-resolve with a recorded reason when
>    fresh evidence later supports every claim.
> 10. **Triage trigger and surface.** Trigger: open items created after
>     the END of the last triage attempt (so ask_user's own items never
>     retrigger the pass that created them; a failed pass never
>     self-retries — the next new item starts one). Surface: the brief's
>     steer/stop/ask_user/resolve_attention PLUS `run_checks`,
>     `review_completion`, `list_attention`, `read_records`,
>     `worker_status`, `list_workers` — verification tools, because a
>     triage that cannot check evidence could only escalate or
>     rubber-stamp.
> 11. **Confidence numbers** (engine constants, gold-tested): strong ≥80,
>     steady ≥45; caps — required check missing/failed 30, failed session
>     30, contradiction/lane violation 40, stale worker 55, audit
>     unavailable 60, stale evidence 65, no fresh evidence 70 (worker) /
>     75 (project, "records alone"); blocked worker caps project ≤60,
>     draining worker ≤72. States come from caps and drain flags, never
>     bands alone; a low-but-stable score is STEADY ("little recorded or
>     evidenced yet") — found live: a fresh project must not read
>     "draining — needs eyes soon" while everything observable is
>     healthy.
> 12. **Project gauge scope.** Stopped/failed workers with a reviewed (or
>     absent-but-resolved) digest and no open attention drop out of the

## Build

- Monitor loop (`src/daemon/monitor.ts`, default 30s tick, **zero LLM calls**):
  staleness (`last_message_at` beyond threshold while running), detective lane
  audit (diff vs `base_sha` ∪ porcelain → `core/lanes/lane-check` →
  `lane_violation` + `contradicted` evidence signal), evidence freshness
  (`evidence_runs.head_sha` vs worktree HEAD → stale), completion-claims scan
  (done-claims without passing evidence → `unsupported_claim`). Reuse the
  ported git-truth observer (`src/adapters/git/runner.ts`) — do not shell out
  ad hoc.
- Check runner (`adapters/checks/run-checks.ts`), written fresh: run a
  worker/project's configured check commands (typecheck, lint, test, build)
  via execFile in the right worktree, capture status + concise summary + a
  log file path, and write `evidence_runs` rows keyed to the worktree's
  `head_sha` at run time (that key is what makes staleness detectable).
  Exposed as the `run_checks` manager tool.
- Confidence engine (`core/confidence/engine.ts`), written fresh and **pure**
  (typed inputs in, explainable report out — no I/O): implement the
  architecture §9 semantics exactly, including the cap mechanism (positive
  signals accumulate a score; trust-critical failures apply hard caps that a
  pile of positives cannot overcome) and per-score debug output (applied
  caps + signals with reasons). Build the §9 gold-scenario regression suite
  alongside it — those scenarios are the specification; if a scenario is
  ambiguous, sharpen it with the user rather than guessing.
- New thin evidence adapter: SQLite rows (evidence_runs, completion_digest
  claims, lane audits, worker liveness) → engine evidence signals. Link each
  digest claim to its evidence run so claims resolve to
  verified/unverified/contradicted.
- Attention queue UI: the one loud surface (vision) — open items ranked by
  priority, resolve/dismiss actions, matching existing UI conventions.
- Event-driven manager triage (jobs kind `triage`): runs only when new open
  attention items exist since last triage, batched. **Never on Darwin's main
  session** — run it like distillation, on a fork (or fresh session seeded
  with the attention batch + records) with `GALAPAGOS_TRIAGE_MODEL`
  (default `claude-haiku-4-5`; the user can raise it). Triage can
  steer/stop workers, `ask_user`, and `resolve_attention`. **Management by
  exception:** clean completions are auto-reviewed
  (`completion_digests.status = manager_reviewed`); only failures,
  contradictions, and direction calls escalate to the user.
- Per-worker and project confidence gauges rendered on `/` and `/workers`
  (one bar each; caps/signals in a debug drilldown only).

## Forbidden

No per-tick LLM calls, no triage on Darwin's main session, no new record
types, no default sub-bars for confidence dimensions, no confidence input
from how confident agent prose sounds.

## Exit criterion

Gauges move for real reasons: a stale worker drains confidence, a failing
required check blocks it, and a worker's question reaches the user in chat
via triage — while a clean completion passes without interrupting the user.
