# Chunk 4 — Monitoring, evidence, confidence

> Brief reworked after Chunk 1's completion (2026-07-04); supersedes the
> original scaffold brief.

You are implementing Chunk 4 of Galapagos. Read `docs/vision.md` and
`docs/architecture.md` first — they are the binding contract and override this
brief on any conflict. Consult the Claude Agent SDK docs — do not guess SDK
APIs.

## State when you start

Chunks 1–3 done: Darwin chat (subscription auth via the user's installed
Claude binary; live turns verified from the user's terminal only), per-project
records store with fork-based distillation, workers streaming in lanes from
worktrees under `<GALAPAGOS_STATE_DIR>/worktrees/`, completion digests parsed,
`attention_items` table populated by lane-stop checks and unstructured
completions (no queue UI yet, no monitor loop). Central SQLite, everything
project_id-scoped. Tests via `tsconfig.node.json` → `node --test`.

## Build

- Monitor loop (`src/daemon/monitor.ts`, default 30s tick, **zero LLM calls**):
  staleness (`last_message_at` beyond threshold while running), detective lane
  audit (diff vs `base_sha` ∪ porcelain → `core/lanes/lane-check` →
  `lane_violation` + `contradicted` evidence signal), evidence freshness
  (`evidence_runs.head_sha` vs worktree HEAD → stale), completion-claims scan
  (done-claims without passing evidence → `unsupported_claim`). Reuse the
  ported git-truth observer (`src/adapters/git/runner.ts`) — do not shell out
  ad hoc.
- Check runner (`adapters/checks/run-checks.ts`), written fresh: run a
  worker/project's configured check commands (typecheck, lint, test, build)
  via execFile in the right worktree, capture status + concise summary + a
  log file path, and write `evidence_runs` rows keyed to the worktree's
  `head_sha` at run time (that key is what makes staleness detectable).
  Exposed as the `run_checks` manager tool.
- Confidence engine (`core/confidence/engine.ts`), written fresh and **pure**
  (typed inputs in, explainable report out — no I/O): implement the
  architecture §9 semantics exactly, including the cap mechanism (positive
  signals accumulate a score; trust-critical failures apply hard caps that a
  pile of positives cannot overcome) and per-score debug output (applied
  caps + signals with reasons). Build the §9 gold-scenario regression suite
  alongside it — those scenarios are the specification; if a scenario is
  ambiguous, sharpen it with the user rather than guessing.
- New thin evidence adapter: SQLite rows (evidence_runs, completion_digest
  claims, lane audits, worker liveness) → engine evidence signals. Link each
  digest claim to its evidence run so claims resolve to
  verified/unverified/contradicted.
- Attention queue UI: the one loud surface (vision) — open items ranked by
  priority, resolve/dismiss actions, matching existing UI conventions.
- Event-driven manager triage (jobs kind `triage`): runs only when new open
  attention items exist since last triage, batched. **Never on Darwin's main
  session** — run it like distillation, on a fork (or fresh session seeded
  with the attention batch + records) with `GALAPAGOS_TRIAGE_MODEL`
  (default `claude-haiku-4-5`; the user can raise it). Triage can
  steer/stop workers, `ask_user`, and `resolve_attention`. **Management by
  exception:** clean completions are auto-reviewed
  (`completion_digests.status = manager_reviewed`); only failures,
  contradictions, and direction calls escalate to the user.
- Per-worker and project confidence gauges rendered on `/` and `/workers`
  (one bar each; caps/signals in a debug drilldown only).

## Forbidden

No per-tick LLM calls, no triage on Darwin's main session, no new record
types, no default sub-bars for confidence dimensions, no confidence input
from how confident agent prose sounds.

## Exit criterion

Gauges move for real reasons: a stale worker drains confidence, a failing
required check blocks it, and a worker's question reaches the user in chat
via triage — while a clean completion passes without interrupting the user.

## Verification

- `npm test` green: ported engine regression suite (identical semantics),
  evidence-adapter unit tests (row → signal mapping), monitor-tick unit tests
  against fixture state.
- Manual (user, own terminal): kill a worker mid-task → stale → attention →
  triage escalation; complete a clean scoped task → digest `manager_reviewed`,
  no user interruption.
